      • 暂存区
        工作区有一个目录.git，是git的版本库
        git的版本库有很多东西，重要的有：比如
        1 vim HEAD
        内容如下 ref: refs/heads/master,再查看refs/heads/master,
        b8572f3352f77459e66aa1272caf71716c8c3366 ,这不就是b8572f3352f77459e66aa1272caf71716c8c3366 append GPL版本么？
        2 stage（或者叫index）的暂存区
        前面讲了我们把文件往Git版本库里添加的时候，是分两步执行的：
        1）git add把文件添加进去，实际上就是把文件修改添加到暂存区；
        2)git commit提交更改，实际上就是把暂存区的所有内容提交到当前分支。
        因为我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，所以，现在，git commit就是往master分支上提交更改。
      • 管理修改
        为什么说Git管理的是修改，而不是文件呢？验证如下：
        1）加一行内容

      2）添加：

      3)再修改readme.txt

      4)提交：

      5)提交后，再看看状态
      发现第二次的修改没有提交！！！
      操作步骤如下：
      第一次修改 -> git add -> 第二次修改 -> git commit
      前面讲了，Git管理的是修改，当你用git add命令后，在工作区的第一次修改被放入暂存区，准备提交，但是，在工作区的第二次修改并没有放入暂存区，所以，git commit只负责把暂存区的修改提交了，也就是第一次的修改被提交了，第二次的修改不会被提交。
      那怎么提交第二次修改呢？
      第一次修改 -> git add -> 第二次修改 -> git add -> git commit

      • 删除文件
        1) 先新加一个文件

      2)把工作区的 test.txt 删掉rm test.txt
      3)git status 告诉你哪些文件删除了

      4)如果确定要从版本库删除，那么用git rm,然后git commit

      5)另一种情况：删除了，要从版本库恢复
      git checkout -- test.txt
      git checkout 用版本库里的版本替换工作区的版本
