容器部署：裸机，虚机哪个更好？

Veröffentlicht am 2017-12-14

翻译自：http://containerjournal.com/2017/01/25/better-containers-bare-metal-virtual-machines/

Docker部署时，一个重要的问题：部署在裸机上还是虚机？

  • 裸机的优势：
    • 更好的性能。性能的测试可以看看这篇性能测试
    • 能访问特殊设备
    • 相对于裸机的非容器化app，更具移植性(protable)
  • 裸机的劣势（是不是就是虚机的优势？）：
    • 相对于非容器化的App，还是有性能损失
    • 缺少隔离性，vm有一层重型的hypervisor隔离租户和宿主机，但这并不意味着docker就不安全或者不够稳定。
    • 相对于vm（具备instance复制），需要重新安装docker，或从habor重新拉取image，尽量这并不难，但是额外增加了操作， 相对缺少移植性。
  • 总结：视情况而定

承接上面五个字(It depends),这是另外一篇相关的讨论,如下。
任何IT的决定都由n个因素决定：Performance, scalability, reliability, security, existing systems, current skillsets, and cost （仅举这几个)。所以，当有人决定要在生产环境上容器时，这些因素都应考虑(merit consideration)
docker 允许相关人员无视底层的差异无缝（seamlessly）部署applications，不管是裸机还是VM,不管是Datacenter 还是 公有云。但是往哪里部署，这就视情况而定啦。
关键是，不要被其中任何一个所局限。你可以轻而易举地从这移植到那里。但是这么多选择，你就得好好斟酌了，这不仅取决于你现在做的，还取决于你将来用它来做什么。
所以，我无法确切回答你“应该哪哪部署app”的问题，不过我可以提供一些因素帮助你做决定。
Latency:不能容忍延迟的应用，放在裸机上更好。这适用于金融行业（例如：交易系统就是个典型例子）
Capacity:VM 通过优化系统负载来（make their bones），如果你的容器化应用不会消耗物理机上的所有容量，虚机可能更好。
Mixed Workloads:裸机只能运行一个os实例，如果你想在同一台机器上跑Windows和Linux 容器，虚机更好。

Mesos：数据中心的细粒度资源共享平台

Veröffentlicht am 2017-12-14

Abstract
我们推出mesos，一个可以在不同计算框架(computing frameworks)（例如hadoop，MPI）之间，共享资源的平台。共享资源能提交提高集群的资源利用率，避免每个应用数据复制。Mesos以细粒度的方式来资源共享，允许多种框架通过顺序读取每台机器上的数据来实现数据本地化(data liocality)。为支持现在frameworks复杂的调度，Mesos提出一种叫资源供给(resource offers)的分布式两层级调度机制。Mesos决定给每一个框架多少资源，同时，框架决定接受哪个资源以及将特定计算分配给特定资源。我们的结果显示Mesos可以在有不同框架的集群中实现近似最佳的数据本地化，集群的节点可以扩展到50000个，并实现HA。

Golang内存模型

Veröffentlicht am 2017-12-14

Introduction

Golang的内存模型定义了这样一种情况：在一个goroutine里写入的变量，能够保证被另一个goroutine观测到。

Advice

修改数据能够同时被多个goroutine访问，这样的程序必须串行访问，以防止数据不一致。
为串行访问，保护数据。方式有管道操作和其他在sync或者sync/atomic包下面的同步原语

Happens Before

在单个goroutine内部,读写数据由程序指定先后顺序。也就是说，在单个goroutine,编译器和进程会记录读写执行顺序,这种情况只会在重排序(reordering)没有改变既有的动作下发生。
因为重排序，每个goroutine的执行顺序可能不一样。例如：一个goroutine 执行

另一个则可能

为明确读写顺序的要求，我们定义了happens before，一个内存操作中的局部排序。如果 e1 发生在 e2 之前，那么我们说 e2 发生在 e1 之后。同样的，如果e1 不早于 e2也不晚于 e2，那么 e1 和 e2 并发。
在单个goroutine，happens before的顺序由程序指定。
如果满足下面的条件，那对变量v的写w，能够被 读r 所观测到。

  1. 读的发生不早于写
  2. 没有其他的写发生在改写之后和该读之前。
    为确保对v的写能被读观察到，必须保证该写是被读观测到的唯一写。也就是说，读被保证了，如果满足以下条件：
  3. 写在读之前
  4. 任何其他对共享变量v的写发生在w之前或者r之后
    这一对条件的约束性要强于上一对。它要求在执行写或者读时，不能有其他的写。
    在单个goroutine，没有并发。所以以上两对定义是等价的。r读最近w所写的。当多个goroutines访问共享变量v，他们必须用同步原语来建立happens-before条件，保证读到所希望的写入。

操作步骤如下：
第一次修改 -> git add -> 第二次修改 -> git commit
前面讲了，Git管理的是修改，当你用git add命令后，在工作区的第一次修改被放入暂存区，准备提交，但是，在工作区的第二次修改并没有放入暂存区，所以，git commit只负责把暂存区的修改提交了，也就是第一次的修改被提交了，第二次的修改不会被提交。
那怎么提交第二次修改呢？
第一次修改 -> git add -> 第二次修改 -> git add -> git commit
